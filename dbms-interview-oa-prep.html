DBMS Basics & Keys

Foundation questions — har interview yahin se shuru hota hai.

DBMS vs RDBMS

DBMSRDBMS
Data files/folders mein store — no relation enforcedData tables mein store, rows/columns ke saath relations
ACID guarantee nahiACID properties follow karta hai
No normalization conceptNormalization support karta hai
Example: file system, XML DBExample: MySQL, PostgreSQL, Oracle

Types of Keys

  • Primary Key: uniquely row identify karta hai, NULL allow nahi karta
  • Candidate Key: saare attributes jo primary key ban sakte hain
  • Super Key: attributes ka set jo row uniquely identify kare (extra attributes ke saath bhi)
  • Foreign Key: doosri table ke primary key ko reference karta hai — referential integrity maintain karta hai
  • Composite Key: do ya zyada columns milke primary key banate hain
  • Alternate Key: candidate keys jo primary key nahi bane

OA gotcha: "Har Super Key Candidate Key nahi hoti, but har Candidate Key Super Key hoti hai" — ye line yaad rakho.

Constraints

  • NOT NULL — column empty nahi ho sakta
  • UNIQUE — duplicate values allow nahi
  • CHECK — condition enforce karta hai
  • DEFAULT — value na di jaaye toh default use ho
  • PRIMARY KEY — NOT NULL + UNIQUE
  • FOREIGN KEY — referential integrity
TABLE 03 · core

ER Model

Entity-Relationship model real world ko entities, attributes, aur relationships mein todta hai — database design ka pehla step.

Core Components

  • Entity: real-world object (e.g. Student, Course) — Entity Set uski collection
  • Attribute: entity ki property (simple, composite, derived, multi-valued)
  • Relationship: do entities ke beech association (e.g. Student "enrolls" Course)
  • Cardinality: relationship mein participation ka count — 1:1, 1:N, M:N

Student aur Course ke beech M:N relationship hai — isliye Enrollment ek junction/bridge table banti hai, jisme dono ke foreign keys hote hain.

Weak vs Strong Entity

Strong EntityWeak Entity
Apni khud ki primary key hoti haiKhud ki primary key nahi — kisi strong entity pe depend karti hai
Example: StudentExample: Dependent (employee ke bina exist nahi karta)

Generalization vs Specialization

Generalization: bottom-up — multiple entities ke common features ko combine karke ek higher-level entity banana (e.g. Car + Bike → Vehicle). Specialization: top-down — ek entity ko specific sub-entities mein todna (e.g. Employee → Manager, Developer).

TABLE 04 · core

Normalization

Redundancy hatane aur data integrity maintain karne ke liye tables ko structured way mein todna. OA mein "is table ka highest normal form batao" type questions bahut aate hain.

Normal Forms — step by step

FormRuleRemoves
1NFHar column atomic value rakhe (no multi-valued/repeating groups)Non-atomic values
2NF1NF + no partial dependency (non-key attribute pura primary key pe depend kare, part pe nahi)Partial dependency
3NF2NF + no transitive dependency (non-key attribute kisi doosre non-key pe depend na kare)Transitive dependency
BCNFHar determinant candidate key ho (3NF ka strict version)Remaining anomalies jo 3NF miss kar deta hai

Functional Dependency — quick recall

A → B ka matlab: A ki value se B ki value uniquely determine ho jaati hai. Example: student_id → student_name.

Partial Dependency: non-key attribute composite key ke sirf ek part pe depend kare. Transitive Dependency: A→B aur B→C ho, toh A→C transitive hai (agar B key nahi hai).

Denormalization

Jaanbujh kar redundancy add karna — reads fast karne ke liye, joins kam karne ke liye. Trade-off: write anomalies aur storage badh sakta hai. Read-heavy systems (reporting, analytics) mein common hai.

SQL & Joins

Query-writing questions OA mein directly aate hain — joins, aggregation, aur subqueries pe strong pakad zaroori hai.

Joins compared

JoinReturns
INNER JOINSirf matching rows dono tables mein
LEFT JOINLeft table ki saari rows + matching right rows (na mile toh NULL)
RIGHT JOINRight table ki saari rows + matching left rows
FULL OUTER JOINDono tables ki saari rows, unmatched pe NULL
SELF JOINTable apne aap se join (e.g. employee-manager hierarchy)
CROSS JOINCartesian product — har row har row se pair

WHERE vs HAVING — classic gotcha

SELECT dept, COUNT(*) as emp_count
FROM employees
WHERE salary > 50000      -- row-level filter, GROUP BY se pehle
GROUP BY dept
HAVING COUNT(*) > 5   -- group-level filter, GROUP BY ke baad;

WHERE individual rows filter karta hai grouping se pehle. HAVING groups ko filter karta hai aggregate function ke result pe — WHERE mein aggregate function use nahi kar sakte.

Common Query Patterns

-- 2nd highest salary
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- duplicate rows dhoondhna
SELECT email, COUNT(*)
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

TABLE 06 · core

Transactions & ACID

Transaction ek logical unit of work hai — poori ki poori execute hogi ya bilkul nahi.

ACID Properties

  • Atomicity: transaction ke saare operations ek unit hain — sab ya kuch nahi (all-or-nothing)
  • Consistency: transaction database ko ek valid state se doosri valid state mein le jaaye
  • Isolation: concurrent transactions ek dusre ko interfere na karein, jaise sequentially chal rahe hon
  • Durability: commit hone ke baad changes permanent hain, chahe system crash ho jaaye

Transaction States

Active → Partially Committed → Committed, ya kisi bhi point pe Failed → Aborted. Active se failure hone pe seedha Failed state mein jaate hain, jahan se Aborted (rollback) hota hai.

Isolation Levels (weakest → strongest)

LevelAllows
Read UncommittedDirty reads possible
Read CommittedDirty read prevent, but non-repeatable read possible
Repeatable ReadNon-repeatable read prevent, but phantom read possible
SerializableSab anomalies prevent — sabse strict, sabse slow

Dirty Read: uncommitted data padh lena. Non-repeatable Read: same row do baar padhne pe different value. Phantom Read: same query dobara chalane pe naye rows dikhna.

TABLE 07 · core

Concurrency Control

Multiple transactions ek saath chalte hain — data consistency banaye rakhne ke liye control mechanisms chahiye.

Lock-Based Protocols

  • Shared Lock (S): read ke liye — multiple transactions ek saath le sakte hain
  • Exclusive Lock (X): write ke liye — sirf ek transaction hold kar sakta hai
  • Two-Phase Locking (2PL): Growing phase (locks acquire) + Shrinking phase (locks release) — koi bhi lock release hone ke baad naya lock acquire nahi kar sakte. Serializability guarantee karta hai.

Deadlock in DBMS

Do transactions ek dusre ke lock release hone ka wait karte reh jaate hain — OS deadlock jaisa hi concept. Handle karne ke tareeke: timeout-based detection, aur wait-for graph se cycle detect karke ek transaction ko abort (rollback) karna.

Other Concurrency Techniques

  • Timestamp Ordering: har transaction ko unique timestamp milta hai, usi order mein execute hone deta hai
  • Optimistic Concurrency Control: assume karo conflict nahi hoga — commit time pe validate karo
  • Multiversion Concurrency Control (MVCC): data ke multiple versions rakhta hai — readers writers ko block nahi karte (PostgreSQL, MySQL InnoDB use karte hain)
TABLE 08 · core

Indexing & Storage

Index query performance drastically improve karta hai — but write operations ko slow kar sakta hai. Trade-off samajhna important hai.

Clustered vs Non-Clustered Index

ClusteredNon-Clustered
Actual data rows ko physically sort/store karta hai index order meinSeparate structure — pointer rakhta hai actual row tak
Table mein sirf ek ho sakta haiTable mein multiple ho sakte hain
Usually primary key pe automaticallyManually create karte hain frequently-queried columns pe

B+ Tree — why databases use it

Balanced tree hai jahan saara actual data leaf nodes mein hota hai, aur leaf nodes linked list ki tarah connected hote hain — isse range queries fast hoti hain. Internal nodes sirf navigation ke liye hote hain (keys, no data), isliye ek node mein zyada keys fit hoti hain → tree ki height kam → disk I/O kam.

Kab index use karo, kab nahi

Use karo jab

  • Column frequently WHERE/JOIN mein use hota ho
  • High cardinality (many unique values)
  • Read-heavy tables

Avoid karo jab

  • Table chhoti hai (full scan hi fast)
  • Write-heavy table (index update overhead)
  • Low cardinality column (e.g. boolean)
